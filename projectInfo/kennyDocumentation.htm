1. Why Do We Need Project Kenny top

Word Doc TC Database

  1. Folder structure and Microsoft Word documents that hold product related test cases
  2. Created by Dennis

Benefits

  1. Created the concept of standards for the client and server team
    1. Word Templates
    2. Folder Structure
  2. Relatively easy to enter information into

Limitations

  1. Word as a database
    1. Lacks permissions.
      1. Any user can edit or delete any field and ruin the whole document
    2. Difficult to maintain
      1. Based on styles which are easily corrupted
      2. Numbering system often wrong
    3. Inflexible
      1. Adopted by client and server team minus Bruce. Orca team uses its own standards

Test Case Execution (Excel Spread Sheet)

  1. Microsoft Excel spreadsheets that hold test cases and track their execution status
  2. Created by Dennis

Benefits

  1. Created the concept of standards for the client and server team
  2. Created a way of tracking execution progress for a project

Limitations

  1. Excel as a database
    1. Lacks permissions. Any user can edit any field and ruin the whole spreadsheet
    2. Has been known to lose data without backups
  2. Difficult to maintain
    1. Importing data is a tricky process and time consuming
  3. Limited information
    1. Can only give you the status of the entire project.
    2. Only tracks status by the last ran build.
  4. Inflexible
    1. Adopted by client and server team minus Bruce. Orca team uses its own standards
  5. Lack of environmental information


Kenny Concept (Project Kenny A.K.A Test Case Tracker)

History

  1. Created by Chad and Darren
  2. Designed to improve upon Dennis’s version as well as give Orca a method of tracking

Kenny Objectives

  1. Replacing the excel docs with a system that mainains similar functionality
  2. Replacing the word docs with a system that mainains similar functionality
  3. Web based system that is accessable from anywhere
  4. Better reporting mechanisms for testers, leads, and managers
    1. Plus a reporting system that is extensible
  5. Incorporating Permissions
    1. Guests, tester, lead, admin
  6. More advanced Multi-user execution environment
    1. Multiple users would be allowed to work within the tool simultaneous without hampering other users
  7. Data backup and recovery
  8. Moving all QA groups to the same system to facilitate standards

Limitations

  1. Hasn’t really been tested in a real world environment (Bugs!)
  2. There has been concern that the team would be hesitent to take up a new system. Extra care must be taken to make this system work for the majority of people
  3. Currently there is no plan to tackle the lack of environment information being tracked. Should be

3. QA Process Explanation top

QA Process

  1. Test Case Creation
    1. Large pieces of the product are broken down into components (ex: email, calendar)
      1. Components represented as a folder in the test doc folder
    2. Each functional area of the component is broken down into a category or test procedure (i.e. email!TP001_sendMail)
      1. Test prodcedures are represented as a word document in a component folder
    3. Each category is broken down into many different test cases
      1. Test Cases are represented as rows in the word document
    4. Each test case can be grouped together as different types of testing (ex: functional, boundry, error)
      1. The Orca team groups together tests using keys
      2. It's not clear Client/Server how this team groups tests
  2. Test Case Execution
    1. In order for engineering to state that they have confidence in the product they created a "determined" amount of test cases must be executed
      1. Determining execution
        • Product Marketing
          • Generally Product marketing will determine the acceptable standards that a product must have before it is shipped
        • Once the level of execution has been determined a "team lead" will select which test cases will be executed during the project lifetime.
          • These test cases are currently stored in an excel spreadsheet
    2. Builds
      • Different versions of software are released to QA at periodic times during the "software lifecycle". These versions are known as builds
      • Each build fixes bugs or adds enchancements to the current code base.
    3. Smoke test
      • Smoke Tests are subsets of the general test base which are meant to give the product a "general" limit of stability
      • Smoke Tests are run at the release of every build
    4. Regression Testing
      1. After a sucessful smoketest a subset of the total regression testing (determined in step 1) is run
        • Each bug that is found during this process is filed in bugzilla
      2. On a subsequent build (one where the bug from the last step is fixed) a QA member is in charge of verifying that the fix is valid
        • This is done by running the set of steps used to find the bug or a set of steps defined by the developer
        • QA members are also supposed to "test around" the area of the fix to make sure another bug wasn't created
      3. After a bug is verified QA members must
        • Add a new test case to the testcase documents (if none exist)
        • Add the test case to the tracking doc (not sure if we do this or not)
    5. Final Acceptance Tests
      1. When the product is nearing its release date a set of builds known as "release candidates" or (RC) as produced
        • A subset of test cases known as the "final acceptance tests" (or FAT) are run against each RC build to verify its quality
        • The status of the FAT helps to determine if the final RC candidate is ready to be "shipped"

     

4. How Kenny Works (Tools) top

Current Implementation

  1. Mysql
    1. A MYSQL database is used as the backend of our project. It is a free tool that has good documentation and is easily used
    2. Using version X
  2. PHP
    1. PHP is the primary tool used to create the front end of Kenny. It is a rapid web development tool that interfaces well with MYSQL
    2. Using version X
  3. Microsoft Word
    1. All of the current test cases are contained in microsoft word documents
    2. Each individual test case is a row in the document and each section of the document (results, steps, title) is defined in a style
    3. The idea is to use the parsing tool (see #4) to get the information out of these docs and into the database once and then migrate away from this system
  4. Visual Basic
    1. In order to get the data from the test cases into the test case database a parsing tool was needed
    2. Since Will Camp already created a parsing tool for the client/server tests we decided to adapt his tool for use on the orca docs
    3. My thoughts are that this tool will no longer need to exist once kenny is fully being utilized
